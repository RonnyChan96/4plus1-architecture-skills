# 4+1 架构建模 Skills 使用指南

本文档说明如何在 Codex、Claude Code、Cursor 中试用并推广本项目里的 5 个架构建模规则，以及如何把它们嵌入实际软件架构工作流。

## 1. 这些 skills 解决什么问题

本项目参考公开的软件架构建模实践和华为云 CodeArts Modeling 的 4+1 视图、UML 建模、一致性检查等概念，整理出 5 个面向架构建模工作的 rules/skills。

原始厂商 PDF 不随本仓库分发；如需核对原始材料，请从华为云官方文档渠道获取。

| 能力 | 用途 | 适合场景 |
| --- | --- | --- |
| `$context-model` | 画上下文模型 | 明确系统边界、外部角色、外部系统、接口 |
| `$uml-sequence-diagram` | 画 UML 顺序图 | 梳理关键用例、运行期交互、异常分支 |
| `$4plus1-architecture-view` | 画 4+1 架构视图 | 形成完整架构设计草案或评审材料 |
| `$logic-code-deployment-trace` | 做逻辑-代码-构建-交付-部署追踪 | 检查设计是否落到仓库、构建产物和部署节点 |
| `$architecture-consistency-check` | 做架构一致性检查 | 评审模型是否存在游离元素、关系错误、追踪断点 |

核心思路是：先定义边界，再用场景驱动架构，再追踪到实现和部署，最后做一致性检查。

## 2. 三端目录结构

项目现在维护三套适配格式：

```text
<this-project>/
├── codex-skills/                 # Codex 原生 Skill 格式
├── claude/                       # Claude Code CLAUDE.md + docs 格式
├── .cursor/rules/                # Cursor MDC Rules 格式
├── .gitignore
├── LICENSE
├── NOTICE
└── 使用指南.md
```

许可证：Apache License 2.0。项目不是 Huawei、OpenAI、Anthropic 或 Cursor 官方项目，也不随仓库分发任何厂商原始 PDF 或文档。

三端使用同一套建模方法，但加载机制不同：

| 工具 | 原生识别方式 | 本项目适配目录 |
| --- | --- | --- |
| Codex | `$skill-name` + `SKILL.md` | `codex-skills/` |
| Claude Code | `CLAUDE.md` 项目指令 | `claude/` |
| Cursor | `.cursor/rules/*.mdc` | `.cursor/rules/` |

## 3. 安装到 Codex

当前 Codex skills 位于：

```text
./codex-skills/
```

如果只在当前项目里查看和维护，不需要安装。

如果希望在任意 Codex 会话中通过 `$skill-name` 调用，需要复制到 Codex skills 目录：

```bash
mkdir -p ~/.codex/skills
cp -R ./codex-skills/4plus1-architecture-view ~/.codex/skills/
cp -R ./codex-skills/uml-sequence-diagram ~/.codex/skills/
cp -R ./codex-skills/architecture-consistency-check ~/.codex/skills/
cp -R ./codex-skills/context-model ~/.codex/skills/
cp -R ./codex-skills/logic-code-deployment-trace ~/.codex/skills/
```

复制后新开一个 Codex 会话，再用 `$context-model`、`$4plus1-architecture-view` 等名称调用。

## 4. 安装到 Claude Code

Claude Code 不识别 Codex 的 `SKILL.md` 格式。使用本项目的 Claude 版本时，把 `claude/CLAUDE.md` 和 `claude/docs/architecture-skills/` 复制到目标代码仓根目录：

```bash
cp ./claude/CLAUDE.md <target-project>/CLAUDE.md
mkdir -p <target-project>/docs/architecture-skills
cp -R ./claude/docs/architecture-skills/* <target-project>/docs/architecture-skills/
```

然后在 Claude Code 中直接描述任务，例如：

```text
Use the architecture modeling instructions to draw a context model for an enterprise approval system.
```

Claude 会根据 `CLAUDE.md` 中的路由说明，读取对应的 `docs/architecture-skills/*.md`。

## 5. 安装到 Cursor

Cursor 使用 `.cursor/rules/*.mdc`。把本项目的规则复制到目标代码仓：

```bash
mkdir -p <target-project>/.cursor/rules
cp -R ./.cursor/rules/* <target-project>/.cursor/rules/
```

然后在 Cursor Chat 中直接描述任务，例如：

```text
Draw a 4+1 architecture view for this system and include logic-code-deployment traceability.
```

Cursor 会根据 `.mdc` 规则的描述和当前文件上下文加载相关规则。

## 6. 最小试用方式

Codex 中可以显式输入：

```text
Use $context-model to draw a context model for an online course platform with students, teachers, payment provider, email service, and admin users.
```

或者中文：

```text
使用 $context-model 为在线课程平台画上下文模型：学生购买课程，老师发布课程，支付平台处理付款，邮件服务发送通知，管理员管理内容。
```

如果 skill 已安装并被识别，Codex 会读取该 skill 的建模规则，并输出上下文图、接口表、边界假设和待确认问题。

Claude Code 或 Cursor 中不需要 `$skill-name`，可以说清楚任务类型：

```text
请根据项目架构建模规则，为在线课程平台画上下文模型：学生购买课程，老师发布课程，支付平台处理付款，邮件服务发送通知，管理员管理内容。
```

## 7. 推荐工作流

### 阶段 1：需求和边界澄清

目标：先搞清楚系统是什么，什么不是系统。

使用：

```text
使用 $context-model 根据下面的需求画上下文模型，并列出系统边界、外部角色、外部系统和接口。

需求：
我们要做一个企业审批系统。员工提交报销申请，直属主管审批，财务复核后打款。系统需要对接企业微信、财务系统、银行代付接口和邮件服务。
```

产出应包括：

- 一个上下文图。
- Actor 和 ExternalSystem 列表。
- 接口清单。
- 系统边界假设。
- 需要业务确认的问题。

### 阶段 2：关键场景顺序图

目标：用顺序图暴露隐藏需求、异常分支和运行期约束。

使用：

```text
使用 $uml-sequence-diagram 为“报销申请审批通过并打款”画 UML 顺序图。

参与方：
员工、审批系统、主管、财务系统、银行代付接口、邮件服务。

要求：
包括主管拒绝、财务复核失败、银行打款失败三个异常分支。
```

产出应包括：

- PlantUML 或 Mermaid 顺序图。
- 参与方和模型元素映射。
- `alt`、`opt`、`loop` 等片段。
- 对遗漏异常路径或顺序不确定性的说明。

### 阶段 3：形成 4+1 架构视图草案

目标：把边界、场景、逻辑结构、代码结构、部署结构组织成一组一致的架构视图。

使用：

```text
使用 $4plus1-architecture-view 为企业审批系统生成 4+1 架构视图草案。

已知信息：
- 前端：Web 管理台、移动端 H5
- 后端：审批服务、用户服务、通知服务、支付适配服务
- 数据：审批数据库、审计日志
- 外部系统：企业微信、财务系统、银行代付接口、邮件服务
- 部署：Kubernetes，审批服务和通知服务独立部署

请输出各视图、关键图和一致性检查清单。
```

产出应包括：

- 用例视图。
- 逻辑视图。
- 开发视图。
- 部署视图。
- 运行视图。
- 视图之间的追踪摘要。

### 阶段 4：追踪到代码、构建和部署

目标：确认架构元素是否真的能落到代码仓、构建产物、交付包和部署节点。

使用：

```text
使用 $logic-code-deployment-trace 为下面架构元素建立逻辑-代码-构建-交付-部署追踪表。

逻辑元素：
- approval-service：审批服务
- user-service：用户服务
- notification-service：通知服务
- payment-adapter：支付适配服务

代码仓：
- approval-backend
- user-center
- notification-worker
- payment-adapter

构建产物：
- approval-service.jar
- user-service.jar
- notification-worker.jar
- payment-adapter.jar

部署：
- Kubernetes namespace: approval-prod
- Deployment: approval-service, user-service, notification-worker, payment-adapter
```

产出应包括：

- `Scenario -> Logical -> Code -> Build -> Delivery -> Deployment -> Runtime` 追踪表。
- Manifest、Build From、Deployed To 等关系说明。
- 缺失链路列表。

### 阶段 5：架构一致性评审

目标：在设计评审或上线前检查模型是否自洽。

使用：

```text
使用 $architecture-consistency-check 检查下面的 4+1 架构模型是否一致。

模型事实：
- 上下文模型里有两个 System：审批系统、支付适配系统。
- Actor 员工直接连到了审批系统，没有通过 Interface。
- approval-service 有代码仓 Manifest。
- notification-service 没有 Manifest 到代码仓。
- approval-service.jar 没有 Build From 到代码目录。
- payment-adapter.jar 已部署到 Kubernetes Deployment payment-adapter。
- notification-worker.jar 没有部署目标。

请按严重级别输出问题、影响和修复建议。
```

产出应包括：

- P1/P2/P3 问题列表。
- 每个问题的证据、影响、修复动作。
- 追踪断点摘要。
- 最小修复顺序。

## 8. 端到端 Demo

下面是一次完整工作流的示例，可以直接复制到 Codex、Claude Code 或 Cursor 分多轮执行。

### 第 1 轮：画上下文模型

```text
使用上下文模型规则为企业审批系统画上下文模型。

业务背景：
员工提交报销申请，主管审批，财务复核，银行代付打款。系统需要对接企业微信获取组织和用户信息，对接财务系统同步凭证，对接银行代付接口执行付款，对接邮件服务发送通知。

请输出：
1. 上下文图
2. 外部参与方表
3. 接口表
4. 边界假设和待确认问题
```

### 第 2 轮：画关键顺序图

```text
基于上一轮的上下文模型，使用 UML 顺序图规则画“报销审批通过并打款”的顺序图。

必须包含：
- 主管审批通过
- 财务复核通过
- 银行代付成功
- 银行代付失败后的重试或人工处理分支
```

### 第 3 轮：生成 4+1 视图

```text
基于前两轮结果，使用 4+1 架构视图规则生成企业审批系统的 4+1 架构视图草案。

架构初稿：
- Web Admin：管理端
- Mobile H5：员工端
- approval-service：审批服务
- user-service：用户服务
- finance-adapter：财务系统适配服务
- payment-adapter：银行代付适配服务
- notification-service：通知服务
- PostgreSQL：业务数据库
- Kafka：异步事件
- Kubernetes：生产部署环境
```

### 第 4 轮：建立追踪表

```text
使用逻辑-代码-部署追踪规则将上一轮逻辑元素追踪到代码、构建产物和部署对象。

代码仓和部署信息：
- approval-service -> repo approval-backend -> approval-service.jar -> Deployment approval-service
- user-service -> repo user-center -> user-service.jar -> Deployment user-service
- finance-adapter -> repo finance-adapter -> finance-adapter.jar -> Deployment finance-adapter
- payment-adapter -> repo payment-adapter -> payment-adapter.jar -> Deployment payment-adapter
- notification-service -> repo notification-worker -> notification-worker.jar -> Deployment notification-worker

请输出追踪表和缺口。
```

### 第 5 轮：做一致性检查

```text
使用 $architecture-consistency-check 对上述 4+1 视图和追踪表做一致性检查。

请重点检查：
- 上下文模型是否只有一个 System
- Actor/ExternalSystem 是否都通过 Interface 交互
- Service/MicroService/Component 是否都有 Manifest 到代码
- Build target 是否都有 Build From
- Delivery/Exec target 是否都有 Deployed To
- 运行视图是否引用已有逻辑元素，而不是新建逻辑元素
```

## 9. 团队落地建议

建议把这些 skills 放到 3 个实际工作节点里：

| 工作节点 | 推荐 skill | 产出 |
| --- | --- | --- |
| 需求澄清/方案启动 | 上下文模型、UML 顺序图 | 边界图、关键场景图 |
| 架构设计 | 4+1 架构视图、逻辑-代码-部署追踪 | 4+1 视图、追踪表 |
| 架构评审/上线前检查 | 架构一致性检查 | 问题清单、修复建议 |

不要一开始就要求 Codex 直接“画完整架构”。更稳妥的方式是从边界和关键场景开始，让逻辑视图、代码视图、部署视图都能被场景验证。

## 10. 常见问题

### 为什么要先画上下文模型？

因为上下文模型定义系统边界。边界不清时，后面的逻辑模块、代码仓、部署节点很容易把外部系统误建成内部组件。

### 顺序图什么时候需要画？

当一个用例涉及多个服务、外部系统、异步消息、重试、异常分支或并发行为时，就应该画。简单 CRUD 不一定需要。

### 4+1 视图一定要全部画吗？

不一定。实际工作中可以按风险选择深度。但至少应明确用例/上下文、逻辑视图、开发视图、部署视图之间的追踪关系。

### 一致性检查是不是只检查图好不好看？

不是。它主要检查模型是否自洽，例如元素是否游离、关系类型是否错误、逻辑元素是否能追踪到代码、构建产物是否能追踪到部署目标。

### 这些 skills 能直接替代架构师判断吗？

不能。它们的作用是让 Codex 按固定建模规则辅助生成、检查和追踪。架构决策仍然需要结合业务目标、技术约束、团队能力和演进成本判断。
